## Refactoring Ideas

- **README.md**: Should contain a description of the package, how to install, example use case(s), authors, etc. 
- **LICENSE**: Looks fine. May be updated down the line
- **environment.yml**: Stores all dependencies required to run everything in the GitHub repository. This may be automatically generated by running the command `conda env export > environment.yml` from `GALCEM/`
- **.gitignore**: Put paths to files we don't want to include in the repo
- **setup.py**: This defines package metadata for distribution on PyPI. For example, we want this to tell PyPI that when a user does `pip install galcem`, it should compile the interpolants and save the models locally. I think this may absorb some of `galcem/prep`; the rest of `galcem/prep` should probably left as input defaults to different functions.  
- **uses/**: Examples, figures, etc. that may be generated by using the `galcem` package. For example, uses will include a script/notebook that generates interpolation fit statistics and figures. The functions for fitting and interpolants, computing their metrics, an creating their plots will be somewhere in the `galcem/` directory. This will allows users who download the `galcem` package to compile the interpolants on their machine. However, we want the use of the package (in this case calling the function to plot interpolants and compute their metrics) to be outside `galcem/`.
    - **uses/output_data/**: Say we have the file `uses/do_something.py` which generates a figure using functions from `galcem`. Then we may save the figure and any other relevant data (e.g. a pandas DataFrame of statistics) to `uses/output_data/do_something/`. Better yet, we may only include Jupyter Notebooks in `uses/` so all the output is self-contained within the notebooks and we would then not need the `/output_data` sub-directory. 
- **test/**: unittests to test functions / classes in `galcem`. Later we can configure these to automatically be run every time someone pushes a new commit. This ensures changes do not break the functionality of `galcem`. 
- **data/**: May contain the original data e.g. downloaded from authors' websites for related papers. A script (perhaps also in `data/`) can then parse this original data into nice formats such as pandas DataFrames. This parsed data that will be used in `galcem` can be put into `galcem/data/` so the only data distributed to the user when they download the package will already be parsed and quick to read in. 
- **galcem/**: The main package that will ultimately get distributed to the user when they install the package via e.g. `pip install galcem`. This should contain functions / classes required to complete desired use cases. 
    - **galcem/onezone.py**: The driver file which contains functions to perform common use cases
    - **galcem/__init__.py**: Defines the package API i.e. how functions may be imported from `galcem`. Should not automatically run the driver/main function for the user, otherwise it will be run every time a user imports the package. 
    - **galcem/classes/**: Objects used throughout drivers/main functions to complete use cases
    - **galcem/data/**: Parsed data e.g. pandas DataFrames which can be used to fit interpolants. 
    - **galcem/models/**: Surrogate models e.g. Mass-Lifetime interpolant that will be used throughout the onezone driver

## General Ideas
- don't use global variables since they can cause confusing errors for a user trying to use a variable with the same name. Better to make these class attributes or pass them as inputs to functions. 
- the `__init__.py` files in `galcem/` just define the API, they shouldn't run anything. 
- git repos shouldn't include binary outputs e.g. PDFs since they are quite large and can grow the repo to be very large after just a few commits. git LFS is a good solution, but once everything is reproducible we may just ignore the pdfs from the repo and tell the interested user how to reproduce the plots.
